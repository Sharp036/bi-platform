Сохранённые запросы

Запросы можно сохранить для повторного использования в виджетах

Зачем сохранять?

  • Повторное использование в нескольких виджетах
  • Централизованное управление SQL
  • Отслеживание количества выполнений
  • Быстрый доступ из дизайнера

Параметризованные запросы

SQL с подстановкой значений из параметров дашборда

SELECT product, SUM(amount)
FROM sales
WHERE region = :region
  AND date >= :startDate
  AND date <= @endDate
GROUP BY product

Подсказка: Используйте :paramName в SQL — значения подставятся из параметров дашборда автоматически.

Палитра компонентов

Нажмите на тип виджета для добавления на канву. Виджет появится в следующей свободной строке.

Канва (Canvas)

Сетка из 12 колонок. Виджеты размещаются по координатам X, Y с указанием ширины W и высоты H.

Панель свойств

Настройка выбранного виджета: заголовок, размеры, источник данных, тип диаграммы, форматирование.

Шаги создания дашборда

Создайте новый отчёт

Reports → New Report. Откроется дизайнер с пустой канвой. Введите название отчёта в toolbar.

Добавьте виджеты

Нажимайте на типы виджетов в палитре слева: Chart, Table, KPI, Text, Filter, Image. Каждый виджет появится на канве.

Привяжите данные

Выберите виджет → в свойствах укажите Data Source: сохранённый запрос или inline SQL с выбором базы данных.

Настройте визуализацию

Выберите тип диаграммы (Bar, Line, Pie...), формат чисел, цвета, размеры виджетов на сетке.

Добавьте параметры

Settings → Parameters. Создайте фильтры (дата, регион и т.д.). Привяжите их к SQL-запросам через :paramName.

Сохраните и опубликуйте

Ctrl+S для сохранения, затем Publish для публикации. Дашборд станет доступен пользователям.

Привязка данных к виджетам

Способ 1: Сохранённый запрос

Выберите готовый запрос из выпадающего списка «Data Source». Запрос уже привязан к конкретной базе данных.

Saved Query → SQL + DB → Execute → Chart

Способ 2: Inline SQL

Напишите SQL прямо в виджете. Сначала выберите Datasource (базу данных), затем введите SQL-запрос.

Datasource → Raw SQL → Execute → Chart

Способ 3: Без данных

Для виджетов Text и Image данные не нужны — они отображают статический контент.

Пример SQL для KPI

-- Одно значение = одна KPI-карточка
SELECT SUM(amount) as total_revenue
FROM sales
WHERE date >= :startDate

-- Результат: 1 строка, 1 колонка
-- total_revenue: 1250000

Пример SQL для нескольких KPI

-- Каждый виджет = отдельный запрос

-- KPI 1: Revenue
SELECT SUM(amount) FROM sales

-- KPI 2: Orders
SELECT COUNT(*) FROM orders

-- KPI 3: Conversion
SELECT ROUND(
  COUNT(CASE WHEN status='paid'
        THEN 1 END)::numeric
  / COUNT(*) * 100, 1
) FROM orders

Интерактивность

Фильтры, drill-down, действия, аннотации

06

Dashboard Actions — межвиджетные действия

5 типов действий

Cross-Filter

Клик по данным в одном виджете фильтрует другие виджеты. Самый частый сценарий: клик по региону в Pie → таблица показывает только этот регион.

Highlight

Выделение связанных данных в целевых виджетах. Остальные данные становятся полупрозрачными.

Navigate

Переход на другой отчёт при клике. Данные передаются как параметры. Создаётся breadcrumb для возврата.

Open URL

Открывает внешнюю ссылку. Поддерживает шаблоны с подстановкой данных: https://crm.com/client/{client_id}

Drill Replace

Скрывает исходный виджет и показывает дочерний на его месте с фильтром по кликнутому значению. Кнопка «Назад» возвращает к исходному виджету. Идеален для детализации: клик по столбцу графика → таблица с детальными строками.

Пример настройки Cross-Filter

При активации фильтра появляется панель вверху дашборда:

Drill-Down навигация

Как работает drill-down

Настройте действие Navigate

В источнике укажите виджет-триггер и целевой отчёт.

Пользователь кликает по данным

Например, по региону «East» в круговой диаграмме.

Переход на детальный отчёт

Система передаёт значение как параметр целевого отчёта.

Breadcrumb для возврата

Цепочка навигации позволяет вернуться на любой уровень.

Многоуровневый drill-down

Level 1: Sales Overview ├── Pie: region = "East" │ ↓ click │ Level 2: Region Details (region=East) ├── Bar: city = "New York" │ ↓ click │ Level 3: City Details (region=East, city=NYC) └── Table: individual transactions Breadcrumb: ┌─────────────────────────────────────────┐ │ Sales Overview > East > New York │ │ ↑ click ↑ click (current) │ └─────────────────────────────────────────┘

Совет: Стройте иерархию отчётов от общего к частному: Overview → Region → City → Store → Transaction.

Контекстное меню виджета

Меню «три точки»

Каждый виджет с данными (Chart, Table, KPI, Filter) имеет кнопку с тремя точками в правом верхнем углу. Доступные действия:

Посмотреть запрос (View Query)

Открывает модальное окно с SQL-запросом виджета. Кнопка «Копировать» копирует SQL в буфер обмена.

Посмотреть как таблицу (View as Table)

Открывает данные виджета в табличном виде. Полезно для графиков, когда нужно увидеть конкретные числа.

Экспорт из модального окна таблицы

Копировать

Копирует всю таблицу в буфер обмена (TSV-формат). Корректно вставляется в Excel и Google Sheets с сохранением колонок.

CSV

Скачивает файл .csv (UTF-8 с BOM для корректного отображения кириллицы в Excel).

Excel

Скачивает файл .xlsx с правильными типами данных.

Drill Replace — детализация на месте

Пошаговая настройка

Создайте родительский виджет (Chart)

Например, столбчатая диаграмма «Статистика цен по магазинам» с агрегированными данными.

Создайте дочерний виджет (Table)

Таблица с детальными строками. Запрос должен возвращать полный набор данных (без фильтра по магазину). Расположите виджет на той же позиции, что и родительский.

Скройте дочерний виджет

Через настройки видимости сделайте дочерний виджет скрытым по умолчанию.

Создайте действие Drill Replace

Вкладка «Dashboard Actions» в правой панели дизайнера. Тип: Drill Replace. Source widget = родительский, Target widget = дочерний (выберите из списка).

Укажите поля фильтрации

Source field = имя колонки в данных родителя (например, store). Target field = имя колонки в дочернем виджете.

Как это работает

1. Пользователь видит график: +------------------------------------+ | Статистика цен по магазинам | | 727 642 639 | | ||| ||| ||| 475 460 | | ||| ||| ||| ||| ||| 183 | | Хол DNS Онл Сит Эль *РБТ* | +------------------------------------+ клик на "РБТ" | v 2. График скрывается, появляется таблица: +------------------------------------+ | [<- Назад] РБТ | +----+----------+----------+---------+ | # | Товар | 1C ID | Цена | +----+----------+----------+---------+ | 1 | BEKO... | 01.01... | 50990 | | 2 | Samsung | 01.02... | 43200 | | .. | (183 строки, store = РБТ) | +----+----------+----------+---------+

Пример SQL

Родительский (агрегация):

SELECT store, COUNT(*) AS cnt
FROM price_data
WHERE dt >= :dt_from
  AND dt <= :dt_to
GROUP BY store
ORDER BY cnt DESC

Дочерний (детали, без фильтра по store):

SELECT store, product, onec_id, price
FROM price_data
WHERE dt >= :dt_from
  AND dt <= :dt_to
ORDER BY price DESC

Важно: Дочерний запрос не содержит WHERE store = ... — фильтрация по кликнутому значению происходит автоматически на клиенте.

Лучшие практики

Проектирование дашбордов

1. Правило «5 секунд»

Ключевые метрики должны быть видны за 5 секунд. Используйте KPI-карточки вверху.

2. Иерархия: от общего к частному

Сверху — итоги (KPI), далее — графики, внизу — детальные таблицы.

3. Не перегружайте

Оптимально 6–8 виджетов. При необходимости — используйте drill-down на детальные отчёты.

4. Единый стиль

Используйте одну цветовую палитру. Консистентные размеры виджетов. Логотип через Overlay.

Оптимизация производительности

5. Оптимизируйте SQL

Используйте индексы, LIMIT, агрегации на стороне БД. Для больших данных — ClickHouse.

6. Параметры вместо множества отчётов

Один параметризованный дашборд лучше 10 статичных. Используйте фильтры и параметры.

7. Используйте сохранённые запросы

Централизованное управление SQL. Легче поддерживать, чем inline SQL в каждом виджете.

8. Настройте расписания

Автоматическая рассылка отчётов снижает нагрузку — не все должны заходить в BI каждый день.

SQL-примеры для разных типов диаграмм

Radar — сравнение продуктов

SELECT metric,
  product_a, product_b, product_c
FROM (VALUES
  ('Quality',  90, 75, 85),
  ('Price',    60, 90, 70),
  ('Support',  85, 65, 80),
  ('Speed',    95, 80, 60),
  ('Features', 70, 85, 90)
) AS t(metric, product_a,
         product_b, product_c)

Heatmap — активность по часам

SELECT
  EXTRACT(HOUR FROM created_at)
    AS hour,
  TO_CHAR(created_at, 'Dy')
    AS day,
  COUNT(*) AS events
FROM user_events
WHERE created_at >= NOW()
  - INTERVAL '7 days'
GROUP BY 1, 2

Funnel — воронка продаж

SELECT stage, users FROM (VALUES
  ('Visitors',     10000),
  ('Sign-ups',     3200),
  ('Trial',        1800),
  ('Paid',         650),
  ('Enterprise',   120)
) AS t(stage, users)

Sankey — потоки трафика

SELECT source, target,
  COUNT(*) AS flow
FROM traffic_events
GROUP BY source, target
ORDER BY flow DESC
LIMIT 50

-- Result:
-- Google    → Landing   → 4500
-- Landing   → Signup    → 1200
-- Landing   → Product   → 2800
-- Product   → Cart      → 900
-- Cart      → Checkout  → 650

Candlestick — биржевые данные

SELECT
  date,
  open_price  AS open,
  close_price AS close,
  low_price   AS low,
  high_price  AS high
FROM stock_prices
WHERE symbol = :symbol
  AND date >= :startDate
ORDER BY date

Treemap — структура расходов

SELECT
  category || ' / ' || subcategory
    AS name,
  SUM(amount) AS value
FROM expenses
WHERE year = :year
GROUP BY category, subcategory
ORDER BY value DESC
